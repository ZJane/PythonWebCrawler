Android touch事件简要笔记

阅读数：2

主要测试下Intercept：

        //ViewGroup dispatchTouchEvent 超级伪码

        if (actionMasked == MotionEvent.ACTION_DOWN) {
            //Down的时候必须走 Jump1
            mGroupFlags & ~FLAG_DISALLOW_INTERCEPT
        }

        //mFirstTouchTarget:ViewParent中消费 事件序列 的子View
        // (从down到up或cancel这一串叫一个事件序列)
        if (actionMasked == MotionEvent.ACTION_DOWN || mFirstTouchTarget != null) {
            final boolean disallowIntercept = (mGroupFlags & FLAG_DISALLOW_INTERCEPT) != 0;
            if (!disallowIntercept) {
                //Jump1: onInterceptTouchEvent
                //在down时return true，决定拦截。下次就跑到了Jump3，不会再调用该方法，以为mFirstTouchTarget为null
                //在down时return false，则随后的move、up都会调用onInterceptTouchEvent
                //        如果随后的move调用onInterceptTouchEvent时返回false，处理事件序列的子View会受到cancel时间
                // 所以滑动冲突，在down时return false，在move时根据 业务逻辑 ，拦截返回true
                intercepted = onInterceptTouchEvent(event);
            } else {
                //Jump2:
                intercepted = false;
            }
        } else {
            //Jump3: ViewGroup上次onInterceptTouchEvent 上次返回true，也就是 本ViewGroup拦截事件后，发送的MOVE、UP等事件
            intercepted = true;
        }
控制的阀门
  1. ACTION_DOWN （Down事件，也就是事件序列的开始）
    • 父View必须要对 是否拦截进行判断 （也就是必然调用 onInterceptTouchEvent）
  2. mFirstTouchTarget （该父View的子View是否 消耗事件序列）
    • 否：证明父View的onInterceptTouchEvent返回true，所以直接拦截，不再调用onInterceptTouchEvent
    • 有：证明子view要消耗事件序列（也就是上次的down的onInterceptTouchEvent返回了false）
  3. disallowIntercept（是否允许父View拦截 事件）
    • 对于Down事件，该flag无效（down时会reset）
    • 对于非Down事件
      • 该flag在TextView、button等普通View中是不会更改设置的，也就是允许拦截。（所以每次都会被调用，这时在onInterceptTouchEvent的down返回false，随后事件都调用 onInterceptTouchEvent，就可以在move中做逻辑判断，如果要父View拦截事件，则return true。下次的move中mFirstTouchTarget==null，所以就不调用onInterceptTouchEvent了）
      • 该flag在滑动冲突的View中，如ViewPager是会更改的disallowIntercept。通过调用requestDisallowInterceptTouchEvent（boolean）,子View的dispatchTouchEvent要重写Down时requestDisallowInterceptTouchEvent（true），这样随后的move 父View走Jump2，不会调用父view的onInterceptTouchEvent。在子view dispatchTouchEvent 的move中做逻辑判断，要让父View处理则requestDisallowInterceptTouchEvent（false），也就是让他走onInterceptTouchEvent。 重写父View的onInterceptTouchEvent，down返回false（让子view的dispatchTouchEvent可以被调用），非down（move）返回true，这样子View在move中requestDisallowInterceptTouchEvent（false），就导致父View onInterceptTouchEvent 返回true，父View拦截成功。

备注:
上面巴拉巴拉的一堆，推荐任玉刚的《Android开发艺术探索》，里面讲的很清楚。

总结

  1. 父View onInterceptTouchEvent down
    • return true，随后事件序列都给父View处理
    • return false，子View也消耗了down事件，则父view默认会一直调用 onInterceptTouchEvent，如果父view在move中onInterceptTouchEvent 返回了true，则子view收到cancel事件，随后事件序列都给父View处理（所以可以解决滑动冲突，因为可以选择父View还是子View来处理事件序列）
  2. 子view操控FLAG_DISALLOW_INTERCEPT，让父View一直走Jump2:，某些情况走Jump1: ，在Jump1: 中父View就可以拦截（这样也可以解决滑动冲突）
